## 1. 摘要

独立开发者最头疼的不是写代码，而是推广。MuiAD 是一个 MCP-first、Self-hosted、AI 驱动的去中心化广告网络，部署在 Cloudflare Workers 上。开发者只需在 AI Agent 里说一句话，AI 就会自动扫描可用广告位、生成推广物料、完成投放和效果追踪。写代码就够了，推广交给 AI。

---

## 2. 背景

我是一名独立开发者，隔三岔五就会搓一个新产品出来。时间久了，手里攒了不少工具网站和小产品。

但问题来了——

**产品做出来了，没人知道。**

我不是不会做推广。AdSense 申请过，被拒了，说内容不够（工具网站嘛，页面就那么几个）。各种广告联盟也研究过，但要么门槛高，要么体验差，要么跟开发者受众完全不匹配。

更深层的问题是：**我本能地抗拒做推广。**

写代码、调 bug，我可以通宵达旦，乐此不疲。但你让我去写推广文案、设计 banner、找合作方、分析投放数据……光是想想就觉得累。

我相信很多独立开发者跟我一样。我们享受创造的过程，但推广这件事，它不在我们的快乐区间里。

所以我想：**能不能让 AI 来做推广？**

不是那种"AI 帮你写个文案"的辅助，而是真正的全自动——AI 自己找推广位、自己生成物料、自己投放、自己优化。我只需要在写代码的时候，顺便跟 AI 说一句"帮我把这个产品推一下"，然后就不用管了。

这就是 MuiAD 的起源。

## 3. 实践过程

### 3.1 从一个被拒的 AdSense 申请说起

一切始于一个很挫败的下午。我的工具网站申请 Google AdSense 被拒了，理由是"内容不足"。

一个工具网站，你让它有多少内容？用户来就是用完就走，不需要读三千字的文章。

但问题不在于 AdSense。问题在于：**现有的广告/推广体系，根本不是为独立开发者设计的。**

- AdSense 需要大量内容
- 联盟营销需要手动找产品、手动加链接
- 广告交换需要人工对接、人工设计物料
- 社交媒体推广需要持续输出内容

每一样都在消耗开发者最稀缺的资源：时间和精力。

### 3.2 核心洞察：MCP 是答案

转折点在于我重新思考了交互方式。

传统的推广工具，不管多"智能"，都需要你打开它的后台、点来点去、配置各种参数。这对开发者来说就是一种心理负担。

但 MCP（Model Context Protocol）改变了一切。

MCP 让 AI Agent 可以直接调用工具。这意味着：**我不需要打开任何推广后台，我在写代码的 IDE 里，顺便就让 AI 把推广的事办了。**

这个洞察决定了 MuiAD 的核心设计：**MCP-first**。

### 3.3 产品设计演进

在 SOLO 的帮助下，我经历了几个版本的设计迭代：

**V1：传统广告服务器**
> 一个 self-hosted 的广告投放系统，类似 Revive Adserver。
>
> 问题：还是需要手动操作，没有解决"不想做推广"的根本问题。

**V2：AI 辅助的广告平台**
> 加了 AI 生成物料的功能。
>
> 问题：还是中心化的，开发者不信任第三方平台拿着自己的流量数据。

**V3：去中心化 + AI 全自动（最终方案）**
> Self-hosted + 去中心化网络 + AI Agent 全自动推广。
>
> 每个开发者部署自己的节点，数据完全在自己手里。节点之间可以互联形成网络。AI 通过 MCP 自动完成所有推广操作。

技术栈选型：

| 组件 | 选择 | 理由 |
|------|------|------|
| 运行时 | Cloudflare Workers | Serverless，免费额度大，全球边缘部署 |
| 数据库 | D1 (SQLite) | CF 原生，SQL 灵活 |
| 缓存 | KV | 广告位配置缓存 |
| 文件存储 | R2 | Banner 图片和 AI 生成的物料 |
| 交互协议 | MCP | AI Agent 直接操作，零推广负担 |

### 3.5 AI Agent 的工作流

这是 MuiAD 的灵魂。开发者只需要说一句话：

```
"帮我把 jsonformatter.pro 推广到所有能触达的开发者受众广告位上"
```

AI Agent 自动执行：

```
📡 扫描网络中的可用广告位
    ↓
🎯 分析哪些广告位适合你的产品（受众匹配、位置、尺寸）
    ↓
🎨 调用 AI 生成 banner 图、宣传文案等物料（多版本 A/B）
    ↓
📤 自动提交到目标广告位
    ↓
📊 持续监控效果，自动优化投放策略
    ↓
🔄 循环寻找新的推广机会
```

### 3.6 三种使用模式

我设计了三种灵活的使用方式，覆盖不同场景：

**孤岛模式**：不连接任何网络，自己的产品互相推广。比如你有工具 A、B、C，A 的页面展示 B 的广告，B 的页面展示 C 的广告，形成自己的流量内循环。

**节点模式**：部署实例，连接到公共网络，与其他开发者互推。你的广告出现在别人的网站上，别人的广告出现在你的网站上。

**免部署模式**：不想部署？直接使用公共节点 all-mui-ad，零基础设施成本。

### 3.7 Credit 积分经济

推广不一定要花钱。MuiAD 内置了积分系统：

- **流量互换**：展示别人的广告赚积分，用积分让别人展示你的广告
- **CPA 付费**：别人给你带来一个注册/下载，你付积分
- **Rev Share**：别人给你带来一个付费用户，你分一部分收入
- **积分转账**：节点之间可以直接互转积分

### 3.8 SOLO 在过程中的角色

整个项目从想法到设计，全程使用 TRAE SOLO 完成：

- **市场调研**：让 SOLO 搜索 AdSense 替代方案、self-hosted 广告服务器、去中心化广告网络等方向
- **竞品分析**：分析 Revive Adserver、Ethical Ad Server、PartnerStack 等现有方案的优劣势
- **产品定义**：通过多轮对话，从 V1 迭代到 V3，逐步明确产品定位
- **架构设计**：技术选型、数据模型、API 设计、项目结构
- **README 撰写**：面向独立开发者的项目文档
- **宣传文稿**：就是你现在看到的这篇

SOLO 不只是帮我写了代码，更重要的是它帮我**思考**。每一轮对话都在逼迫我把模糊的想法变成清晰的设计。

## 5. 效果与总结

### 这个项目教会我的

**1. AI 时代的交互范式变了**

以前做工具，UI 是第一位的。但在 AI Agent 时代，**API 和 MCP 才是第一位的**。用户可能永远不会打开你的 Dashboard，但他的 AI Agent 会每天跟你的 API 交互十次。MuiAD 的设计就是从这个认知出发的——MCP-first，Dashboard 可选。

**2. 去中心化 + AI 是独立开发者的未来**

中心化平台（AdSense、各种联盟）的问题不只是抽成，更关键的是**信任**。开发者不信任别人拿着自己的流量数据。Self-hosted 解决了信任问题，AI 解决了效率问题，去中心化网络解决了规模问题。三者结合，才是一个真正适合独立开发者的推广方案。

**3. Build in public 的力量**

把想法公开出来，让社区参与讨论，这比闭门造车强太多。这篇文稿本身就是 build in public 的一部分。
